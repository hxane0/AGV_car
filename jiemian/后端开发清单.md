# 后端开发清单

## 1. 目标

这份清单按“当前项目实际阶段”整理。

目前你们`还没有开始做 SLAM 建图和导航`，所以和地图、路径、历史轨迹相关的接口，不应再列为当前阶段的必做项。现阶段更现实的目标是：

- 先做 `AGV 状态监控`
- 先让 `AGV 基础状态接口` 跑起来
- 先让 `AGV 详情页` 和后续的状态类页面能联调
- 地图、路径、轨迹留到 SLAM/导航启动后再做

因此后端开发建议按两个阶段推进：

1. `第一阶段：AGV 状态监控`
2. `第二阶段：SLAM / 地图 / 导航能力接入`

---

## 2. 第一阶段：当前必须做的后端能力

这一部分是现在最值得做、也最容易真正落地的。

### 2.1 基础环境

- 启动一个可访问的后端服务，例如 `http://127.0.0.1:8000`
- 提供统一接口前缀：`/api/v1`
- 提供 WebSocket 地址：`ws://127.0.0.1:8000/ws`
- 返回 JSON 数据
- 接口字段名尽量严格按 `ui/api-spec/api.md`
- 如果前端不是和后端同源部署，需要允许跨域访问

### 2.2 必做接口

#### 1. 获取 AGV 列表

- `GET /api/v1/agv`

前端用途：

- AGV 详情页在没有 `id` 时先取第一台车
- AGV 列表/状态监控类页面后续复用
- WebSocket 断开时轮询兜底

最少每台车要返回这些字段：

```json
{
  "id": 1,
  "name": "AGV-001",
  "model": "R550",
  "status": "running",
  "battery": 85.5,
  "position_x": 0,
  "position_y": 0,
  "position_z": 0,
  "orientation": 0,
  "current_task_summary": "A01 -> B03",
  "speed": 0.8
}
```

说明：

- 目前没有建图和导航时，`position_x`、`position_y`、`position_z`、`orientation` 可以先返回默认值，例如 `0`
- 字段建议先保留，后续接入定位后前端不用改接口结构
- 如果当前完全拿不到位置，也建议保留字段但先返回占位值，而不是直接删掉

状态值建议严格使用：

- `running`
- `idle`
- `charging`
- `error`
- `offline`

#### 2. 获取单个 AGV 详情

- `GET /api/v1/agv/{id}`

前端用途：

- AGV 详情页基本信息卡
- AGV 详情页实时状态卡
- 详情页每 2 秒轮询刷新

最少要返回这些字段：

```json
{
  "code": 0,
  "data": {
    "id": 1,
    "name": "AGV-001",
    "model": "R550",
    "status": "running",
    "battery": 85.5,
    "position_x": 0,
    "position_y": 0,
    "position_z": 0,
    "orientation": 0,
    "speed": 0.8,
    "mode": "auto",
    "current_task_id": 42,
    "current_task_summary": "A01 -> B03",
    "communication_quality": "strong",
    "last_update": "2024-01-01T14:32:00Z"
  }
}
```

其中这些枚举值要特别注意：

- `mode`: `auto` / `manual` / `charging`
- `communication_quality`: `strong` / `medium` / `weak` / `disconnected`

#### 3. WebSocket 实时状态

- `ws://127.0.0.1:8000/ws`
- Topic：`agv/status`

前端用途：

- 实时刷新 AGV 状态、电量、速度、模式、当前任务摘要
- 后续地图页恢复真实模式后也可继续复用

前端订阅消息：

```json
{
  "action": "subscribe",
  "topic": "agv/status"
}
```

服务端推送格式：

```json
{
  "topic": "agv/status",
  "data": [
    {
      "id": 1,
      "name": "AGV-001",
      "status": "running",
      "battery": 85.5,
      "position_x": 0,
      "position_y": 0,
      "position_z": 0,
      "orientation": 0,
      "speed": 0.8,
      "current_task_id": 42,
      "current_task_summary": "A01 -> B03",
      "timestamp": "2024-01-01T14:32:00.123Z"
    }
  ]
}
```

还需要支持：

- `ping/pong`
- 断线重连后重新订阅

## 3. 第一阶段内部数据接入

要让上面的接口有真实数据，后端内部至少要打通这些来源。

### 3.1 AGV 实时状态来源

建议接入这些内部数据源：

- `/agv_{id}/battery_state`
- `/agv_{id}/status`

如果现在已经能拿到速度，也可以接入：

- `/agv_{id}/odom`

需要在后端完成的事情：

- 把速度、电量、运行状态、模式、当前任务等信息合并成“当前 AGV 快照”
- 提供给 REST 接口
- 同时推送给 WebSocket

说明：

- 如果当前没有定位结果，不要把“位置能力”当作第一阶段阻塞项
- 可以先把位置相关字段固定为 `0` 或占位值
- 等 SLAM/定位接入后，再把这些字段换成真实数据

---

## 4. 第二阶段：依赖 SLAM / 导航后再做的能力

这部分目前`不是当前必做项`，等你们开始建图和导航后再推进更合适。

### 4.1 地图数据

- `GET /api/v1/map`

后端要做：

- 建立统一地图坐标系
- 提供场景边界
- 提供地图原点与高度基准
- 必要时提供地图模型或底图资源

### 4.2 当前规划路径

- `GET /api/v1/agv/{id}/planned-path`

后端要做：

- 从导航模块获取当前规划路径
- 转成前端所需的 `points`
- 支持无路径时返回空数组

### 4.3 历史轨迹

- `GET /api/v1/agv/{id}/trajectory`

后端要做：

- 持续保存 AGV 历史位置点
- 支持按时间范围查询

最少字段：

- `x`
- `y`
- `z`
- `timestamp`

### 4.4 地图页真实实时状态

当第二阶段开始后，地图页才真正依赖：

- 实时位置
- 朝向
- 地图坐标转换
- 路径叠加显示

---

## 5. 第三阶段：下一批建议做的接口

这部分不是当前 AGV 状态监控跑通的硬依赖，但一做完，系统会完整很多。

### 5.1 Dashboard 总览

- `GET /api/v1/dashboard/summary`
- `GET /api/v1/dashboard/task-trend`
- `GET /api/v1/dashboard/energy`
- `GET /api/v1/dashboard/agv-distribution`

后端要做：

- AGV 在线数统计
- 平均电量统计
- 累计里程统计
- 每日任务趋势统计
- 状态分布统计

### 5.2 事件日志

- `GET /api/v1/events`
- `POST /api/v1/events/{id}/handle`
- WebSocket `events`

后端要做：

- 接收和存储 AGV 相关异常事件
- 支持查询、分页、筛选
- 支持标记已处理
- 支持新事件实时推送

### 5.3 物料检测

- `GET /api/v1/inspections`
- `GET /api/v1/inspections/{id}`
- WebSocket `inspection`

后端要做：

- 接收 `/inspection/result`
- 保存图片地址、颜色、重量、缺陷、置信度
- 支持列表查询与详情查看
- 支持新检测结果实时推送

## 7. 数据库层建议

后端最好尽快确定这些核心表，不然后面接口会越来越乱。

第一阶段建议至少有：

- `agv`：AGV 基础资料
- `agv_latest_status`：每台车最新状态快照
- `task`：任务主表
- `task_log`：任务状态流转日志
- `event_log`：异常/告警事件
- `inspection_record`：检测记录
- `user`
- `audit_log`
- `system_config`

第二阶段再补：

- `agv_trajectory`：历史轨迹点
- `agv_path_snapshot`：当前路径快照
- `map_metadata`：地图元数据

---

## 8. 当前最推荐的开发顺序

建议按这个顺序做：
1. 先做 `GET /api/v1/agv`

2. 再做 `GET /api/v1/agv/{id}`
3. 再做 WebSocket `agv/status`
4. 然后补 Dashboard、事件、检测、任务管理
5. 等 SLAM/导航启动后，再补 `GET /api/v1/map`
6. 再补 `GET /api/v1/agv/{id}/planned-path`
7. 最后补 `GET /api/v1/agv/{id}/trajectory`

这样做的好处是：

- 你可以最快让 AGV 状态类页面从 mock 变成真数据
- 不会被地图和导航能力卡住
- 后面扩展统计、日志、任务管理时不会返工太多

---

## 9. 当前阶段验收标准

后端做到下面这些，就算第一阶段联调成功：

- `GET /api/v1/agv` 能正常返回多台 AGV 状态
- `GET /api/v1/agv/{id}` 能正常返回单台 AGV 详情
- 详情页能显示该车真实电量、速度、模式、当前任务、通信质量
- WebSocket `agv/status` 能实时推送状态变化
- WebSocket 断开后，前端能退回轮询
- 接口字段和 `ui/api-spec/api.md` 保持一致

当前阶段不要求：

- 真正地图坐标
- 路径显示
- 历史轨迹回放

这些属于第二阶段。

---

## 10. 一句话总结

如果只结合你们当前进度，后端现在最先做这 3 个就够了：

1. `GET /api/v1/agv`
2. `GET /api/v1/agv/{id}`
3. WebSocket `agv/status`

而 `GET /api/v1/map`、`GET /api/v1/agv/{id}/planned-path`、`GET /api/v1/agv/{id}/trajectory` 应该放到你们开始做 SLAM 建图和导航之后再做。
